# 工程结构设计

## 目标

项目的核心不是普通聊天，而是“对目标 API 做一组可复用的探针检测，并输出结构化鉴定结果”。  
因此目录结构必须围绕三个对象组织：

1. 检测任务
2. 探针能力
3. 报告展示

## 后端模块职责

### `api/`

只负责 HTTP 暴露，不承载探针业务逻辑。

- `health.py`：健康检查
- `detection.py`：检测任务入口
- `router.py`：统一注册路由

### `core/`

放全局配置和通用基础设施，保持最小化。

- `config.py`：环境变量、应用名、版本、默认超时

### `modules/detection/`

项目核心域，围绕“检测”单一职责组织。

- `schemas.py`：请求体、响应体、探针结果结构
- `service.py`：编排探针执行、汇总信任分、生成报告
- `probes/`：每个探针独立实现，避免一个大文件堆满所有检测逻辑

### `probes/` 的拆分原则

每个探针文件只负责一种检测维度，遵守单一职责：

- `parameter_probe.py`：特性参数兼容性检测
- `response_probe.py`：返回结构、usage、system_fingerprint 等格式检测
- `behavior_probe.py`：行为级测试，例如系统提示词泄露、格式服从性

后续如果要增加：

- `tokenizer_probe.py`
- `tool_call_probe.py`
- `vision_probe.py`
- `context_window_probe.py`

直接在 `probes/` 下增加文件，并在 `registry.py` 注册即可，不需要改动接口层。

## 前端模块职责

前端按“页面 + 功能模块 + 共享组件”拆分，不提前引入复杂状态管理。

### `pages/`

负责页面级拼装：

- `DetectorPage.tsx`：输入 Base URL、API Key、模型名并发起检测
- `ReportsPage.tsx`：展示历史报告或样例报告

### `features/detection/`

围绕“检测”这一前端能力聚合：

- `api.ts`：调用后端检测接口
- `types.ts`：前端使用的检测结果类型
- `components/DetectorForm.tsx`：输入表单

### `shared/components/`

只放真正可复用的壳层和基础组件，避免过早抽象。

## 为什么不用更复杂的架构

当前阶段不需要：

- 微服务
- 前后端共享代码生成
- 消息队列
- 复杂插件系统

原因很简单：现在只有一个核心业务流，“发起检测 -> 执行探针 -> 输出报告”。  
先把探针能力模块化，比提前做技术炫技更有价值。

## 下一阶段建议

建议按这个顺序推进：

1. 先把后端真实 HTTP 调用链打通
2. 再实现 3 个最有辨识度的探针
3. 最后做前端报告可视化和历史记录
